# UC-01 — RDP Brute Force Detection

## 1. Objective

This use case focuses on detecting repeated failed authentication attempts against the Windows endpoint using the Windows Security Event Logs collected in Phase 1.

Phase 0 defines brute force detection as one of the core SOC monitoring objectives of the lab, and Phase 1 establishes the required log pipeline by forwarding Windows Security logs from the monitored endpoint to Splunk. This use case builds directly on that foundation by simulating a brute force attack from the Kali attacker machine and developing a Splunk detection for the resulting telemetry.

---

## 2. Attack Simulation

The attack was performed from the Kali attacker VM against the Windows endpoint over the exposed RDP service on port 3389.

Hydra was used to perform repeated password guessing attempts against the `victim_machine` account:

```bash
hydra -l victim_machine -P password.txt rdp://192.168.100.20 -t 4 -V
```
Command explanation
hydra starts the brute force tool.
- *-l victim_machine* specifies the username being targeted.

- *-P password.txt* tells Hydra to use the password list stored in password.txt.

- *rdp://192.168.100.20* specifies the target protocol and the Windows target IP.

- *-t 4* runs 4 parallel tasks to speed up the attack without making the output too noisy.

- *-V* enables verbose mode so each attempt is shown in the terminal output.

During the attack, multiple authentication attempts were generated in a short period of time, and Hydra eventually identified the valid password for the target account.

## 3. Failed Authentication Events

As established in Phase 1, Windows Event Logs are forwarded into the windows index in Splunk. To review failed authentication activity generated by the brute force attack, the following SPL query was used:
```
index=windows EventCode=4625
| table _time host Account_Name Workstation_Name Source_Network_Address Logon_Type Failure_Reason
| sort - _time
```
Query explanation
- index=windows searches the Windows Event Log data onboarded in Phase 1.
- EventCode=4625 filters for failed logon events.
- table ... selects only the fields useful for initial triage.
- sort - _time sorts the newest failed logons first.

---

### Analyst observations

The results show repeated failed authentication attempts against the same account within a short time period.

Important observations from the raw events:

- **Account_Name** shows the targeted user account: *victim_machine*
- **Workstation_Name** shows the source hostname: *kali*
- **Source_Network_Address** shows the source IP: *192.168.100.10*
- **Logon_Type** is 3, indicating *network-based authentication*
- **Failure_Reason** consistently indicates *invalid credentials*

These repeated failures from the same source form the initial brute force pattern.

## 4. Event Field Analysis

To validate the exact event structure, a single Event ID 4625 record was expanded in Splunk.

The event confirms the most relevant fields for this detection:

- EventCode = 4625 confirms a failed logon event
- Account_Name = victim_machine identifies the targeted account
- user = victim_machine provides the same user context in another parsed field
- Source_Network_Address = 192.168.100.10 identifies the attacking source IP
- Workstation_Name = kali identifies the attacker hostname
- Logon_Type = 3 indicates network logon
- Failure_Reason = Unknown user name or bad password
- Sub_Status = 0xC000006A indicates a bad password condition

This event-level analysis is important because it confirms which fields are actually populated in this lab environment before building the final SPL detection.

## 5. Detection Logic

The following SPL query was used to detect repeated failed login attempts from the same source against the same account within a 5-minute window:
```
index=windows EventCode=4625
| eval user_name=coalesce(user, Account_Name, TargetUserName)
| eval src=coalesce(Source_Network_Address, Workstation_Name)
| search user_name!="ANONYMOUS LOGON"
| bin _time span=5m
| stats count values(Failure_Reason) as failure_reasons values(Workstation_Name) as src_host by _time host user_name src Logon_Type
| where count >= 5
| sort - count
```
Query explanation
- index=windows EventCode=4625 starts with failed Windows logon events.

- eval user_name=coalesce(user, Account_Name, TargetUserName) normalizes the username field because different field names may be populated depending on parsing.

- eval src=coalesce(Source_Network_Address, Workstation_Name) normalizes the source so the query can still work even if one source field is missing.
- search user_name!="ANONYMOUS LOGON" removes obvious noise that is not useful for brute force detection.
- bin _time span=5m groups events into 5-minute windows.
- stats count ... by _time host user_name src Logon_Type aggregates the failed logons by time bucket, victim host, targeted account, source, and logon type.
- where count >= 5 applies a simple detection threshold to identify suspicious repeated failures.
- sort - count shows the highest-volume activity first.

Why this detection works

A single failed login is common in normal environments. This query is designed to look for repeated failed logins clustered in time from the same source against the same account, which is much more consistent with brute force behaviour than ordinary user error.

In this case, the query clearly identifies the victim_machine account being targeted from 192.168.100.10 / kali, with repeated failures grouped inside a 5-minute window.

Phase 0 defines brute force and suspicious authentication monitoring as core design objectives of the lab, and Phase 1 delivers the Windows log pipeline required to support them. This use case is the first practical validation of that design.

## 8. Outcome

The simulated brute force activity from the Kali attacker machine was successfully captured in Splunk using the Windows Security Event Logs onboarded in Phase 1.

The final SPL detection identifies repeated failed authentication attempts against the **victim_machine** account from the same remote source within a short time window. This demonstrates a practical and SOC-relevant brute force detection workflow using Windows Event ID 4625.
